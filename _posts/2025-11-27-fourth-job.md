# SG사 임직원용 총무 관리 프로젝트

## 프로젝트 개요

X 업체와 협력하여 사내대출, 단체보험, 임직원 복지 서비스 위한 이용권 및 전반에 관한 관리

## 프로젝트 기간

2025.02 - 2025.05


## 역할

- 임직원 복지 서비스
  - 이용권 관리
  - 전체 진료 이력 조회
  - 차트 등록


## 어려웠던 점

해당 프로젝트의 핵심 키워드는 **클린 아키텍쳐** 였다. 추후 MSA 구조로 마이그레이션 하는 것을 전제로, 프로젝트간 종속성, 의존성을 최소로 한 채로 
프론트, 백엔드가 전부 개발되어야 했다. 1개의 화면을 구성하기 위해 매번 수십개의 파일을 만들고, 각각의 컨버터를 만들어야 했다. 컨플루언스에 정의된 f/e, b/e, db 관련 사내 컨벤션을 다 읽고 이해하는데만 며칠이 걸렸다.

또한 여러 회사의 인원들, 20명이 넘는 인원을 관리하기 위해 현업이 요구하는 브랜치 전략이 복잡했다. 심지어 클라이언트와 여러 업체간 배포 방식 조율을 위해 중도에 여러 차례 브랜치 전략이 바뀌며 엄청난 혼선을 빗었다.
작은 규모의 팀에서는 사용하지 않았던 git 의 여러 기능을 여러 브랜치간 conflict 을 해결하며 실전에서 익힐 수 있었다.

![sg](assets/img/sg.png)

모든 PR 은 3명의 리뷰어가 검토 후 승인 하는 구조였다. 현업은 클린 아키텍쳐와 일정 수준이상의 전반적인 코드 품질을 꼼꼼하게 검토했다. 소규모의 스프린트 개발에서 했던 당장의 기능 검증 -> 추후 develop 은 용납되지 않았다.
최초 PR 은 12번의 코멘트가 달린 이후에야 통과되었다. 모든 사람이 동일한 과정을 거쳤고, 이로인해 프로젝트 전체적으로 균일한 코드 품질이 유지되었다.


매번 진행되는 프로젝트가 4-5인 이하 규모에서는 기대하기 힘든 구조와 환경이라 배울 점이 많았다. 소규모는 인원이 적어 유연하게 변경 사항에 대해 대응이 가능했지만 시간이 흐를수록 전반적인 코드가 들쑥날쑥하게 되는 경우가 잦았다. 또한 설계에 시간을 충분히 들이지 않아 했던 일을 갈아엎고 또 하게 되는 경우가 빈번했다.

가장 힘들었던 점은 복잡한 비즈니스 요구사항 구현보다는 원활한 의사소통이 잘 이루어지지 않은 것이 힘들었다. 업체 간 알력다툼, 타이트한 일정 내 불합리한 업무 분담으로 인한 팀장과의 불화 등은 마감이 다가올 수록 심해졌다. 종종 업무에 사적 감정을 넣는 팀원들이 생겼고 이는 팀에 부정적인 영향을 미쳤다.
바쁜 와중에 사람들끼리 겪는 충돌 때문에 종종 혼란스러웠지만 말 보다는 자료가 더 도움이 되었다. 다 같이 모인 회의 끝에 엑셀, 컨플, jira 를 활용해 남은 시간에 해야하는 일, 각자 맡아서 소화가 가능한 일, 우선적으로 끝내야 하는 일을 정리했다.
A 가 개발이 끝나야, B 의 개발이 마무리되는 식의 서로 영향을 주고 받는 경우가 많았기 때문에 기능 단위로 개발 인원을 묶어서 빠르게 작업이 가능한 일, 시간이 오래 걸리는 일을 아주 세부적으로 분류했고 팀원들과 공유하는 식으로 점점 일정을 끝내는데 온 정성을 다하기 시작했다.

추상적인 목표가 아니라 구체적으로 달성가능한 목표들이 나열된 이후에는 점점 일정에 뒤쳐지는 일이 줄어들었다. 여전히 시간은 부족했지만 약속한 기간 내 현업에서 요구하는 수준을 맞출 수 있었다.
